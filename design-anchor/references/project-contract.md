# Design Anchor Project Contract

Read this when installing Design Anchor, activating a consumer project, changing source files, editing tokens/specs, or deciding whether a target directory is a consumer project or the Design Anchor source repo.

## Consumer Project Layout

Design Anchor creates and governs these locations:

| Area | Source of truth | Rule |
|---|---|---|
| Components | `src/components/anchor-ui/` | App code imports from `@design` or `@/components/anchor-ui`; do not import implementations from `.anchor` or `node_modules/design-anchor`. |
| Tokens | `src/design-tokens/tokens.json` | Edit this file for product tokens; regenerate CSS after changes. |
| Generated token CSS | `src/styles/design-tokens.generated.css` | Generated output; do not hand-edit. |
| Portal/control plane | `.anchor/` | Holds Portal, schema, rules, sync scripts, demos, and generated governance artifacts. |
| Component specs | `.anchor/src/anchor/schema/components/*.spec.json` | The contract for props, variants, forbidden tags, examples, and AI guidance. |
| Active prompt style | `.anchor/src/anchor/rules/ACTIVE_PROMPT_STYLE.md` | Written by `npx design-anchor theme <prompt.md>`; sync mirrors it into AI coding rules. |
| AI coding rules | `AGENTS.md`, `CLAUDE.md`, `.cursor/rules/anchor.mdc`, `.github/copilot-instructions.md` | Generated by `start`/`sync`; this is how Design Anchor becomes persistent project governance. |
| MCP/hooks | `.mcp.json`, `.cursor/mcp.json`, `.cursor/hooks.json` | Generated integration so tools can inspect tokens/specs and run self-checks. |
| Upstream package | `node_modules/design-anchor/` | Read-only. Update through `anchor upgrade`, never by direct edits. |

## Install Commands

Prefer the package manager already used by the project:

```bash
npm install design-anchor
npx design-anchor start
```

```bash
pnpm add design-anchor
pnpm exec anchor start
```

```bash
yarn add design-anchor
yarn anchor start
```

```bash
bun add design-anchor
bunx design-anchor start
```

Use `start` for activation because it initializes `.anchor`, installs runtime dependencies, writes AI coding rules, configures MCP/hooks, creates visible components/tokens, and opens Portal. Use `portal <tab>` after the project is already initialized.

Do not describe this as onboarding. In the current product model, the skill is the entry point and the npm runtime writes always-on project governance.

## Consumer Vs Source Repo

If `package.json` has `"name": "design-anchor"` and the repo contains `bin/anchor.mjs` plus `src/anchor-portal/`, it is the Design Anchor source package.

In the source package:

- Do not run `npx design-anchor start`; that scaffolds a consumer `.anchor/` into the source repo.
- Edit package source directly, for example `src/components/base`, `src/anchor/schema/components`, `src/design-tokens`, `src/anchor-portal`, `bin`, or `scripts`.
- Run source-package checks such as `npm run sync:anchor`, `npm run anchor:audit`, and `npm run typecheck`.

In a consumer project:

- Business pages belong in the app's normal source tree.
- Component implementation changes belong in `src/components/anchor-ui`.
- Token changes belong in `src/design-tokens/tokens.json`.
- Portal and schema integration belong in `.anchor`.
- Style source extraction uses `npx design-anchor theme <prompt.md>`.
- Ongoing commands should use the npm runtime from the project root: `npx design-anchor sync`, `npx design-anchor audit`, `npx design-anchor portal <tab>`, and `npx design-anchor upgrade`.

## Post-Change Verification

After token, prompt style, or spec edits:

```bash
npx design-anchor sync
```

After UI/component/spec edits:

```bash
npx design-anchor sync
npx design-anchor audit
```

If working inside the Design Anchor source package itself, use source-package scripts instead:

```bash
npm run sync:anchor
npm run anchor:audit
```

Final responses for UI changes must include a `Design Anchor 自检` line.
